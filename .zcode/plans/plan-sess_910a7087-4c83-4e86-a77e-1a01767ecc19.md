# Stage 3 & 4: Page-Layout Gallery + Page-Layout Application

## Approach (per your choices)

- **Gallery location:** Enhance the existing `CreatePageModal.vue` inside the editor ‚Äî add a two-mode picker: "Start blank" vs "Choose layout". Avoids building a whole new page shell.
- **Layout metadata:** Add optional `page_type` + `industry` keys to each of the 12 layouts in `config/designs.php`. Validator already allows extra keys.

This reuses the existing `POST /editor/pages` endpoint (Stage 4), extended to accept an optional `layout_key`, rather than creating a new endpoint.

## 2. Backend: New `BuildPageLayouts` action ‚Äî `app/Actions/Designs/BuildPageLayouts.php`

Returns the 12 layouts as lightweight gallery data (no full block trees, to keep the prop small):

```php
[
  'key' => 'restaurant-home',
  'label' => 'Restaurant Home',
  'page_type' => 'home',
  'industry' => 'restaurant',
  'preview_blocks' => [...],   // first 2-3 blocks for a small thumbnail preview
  'block_count' => 6,
]
```

`preview_blocks` is a trimmed slice (e.g. first 3 nodes) so the modal can render a compact preview without sending all block trees.

---

## 3. Backend: Extend `TenantPageController::store` to accept `layout_key`

**Validation change:** Add optional `layout_key` field:
```php
'layout_key' => ['sometimes', 'string', Rule::in(array_keys(config('designs.page_layouts')))],
```

**Draft config logic:** If `layout_key` is present, deep-clone + regenerate IDs from `config('designs.page_layouts')[$key]['blocks']` (mirrors `ApplySiteKit`'s clone helper). If absent, keep the existing default HeroBlock.

**Extract a shared clone helper:** Move `deepCloneAndRegenerateIds` into a reusable trait or static method so both `ApplySiteKit` and `TenantPageController::store` use the same logic. I'll add it as a static method on `ApplySiteKit` (or a small `CloneBlockTree` action) to avoid duplication.

## 5. Frontend: Enhance `CreatePageModal.vue`

Two-step modal:

**Step 1 ‚Äî Mode selection (always shown):**
- Two buttons: "Start blank" and "Choose layout"
- "Start blank" ‚Üí goes straight to the title/slug form (current behavior)
- "Choose layout" ‚Üí shows the layout grid

**Step 2a ‚Äî Layout grid (when "Choose layout" selected):**
- Filter chips by `page_type`: Home / About / Contact / Menu / Shop / Rooms / Amenities
- Grid of layout cards, each showing a compact preview (`RenderPublicNode` over `preview_blocks`) + label + industry badge
- Clicking a layout selects it and advances to the title/slug form
- "Back" button returns to mode selection

**Step 2b ‚Äî Title/slug form (always shown last):**
- Same title + slug fields as today
- If a layout was chosen, show a "Selected: {label}" indicator with a "Change" link
- The selected `layout_key` is sent with the POST
- `autoGenerateSlug` unchanged

**POST body:** `{ title, slug, layout_key? }` ‚Äî `layout_key` omitted when "Start blank".

## 6. Tests ‚Äî `tests/Feature/PageLayoutApplicationTest.php`

New feature test file:

| Test | What it verifies |
|---|---|
| Store with valid layout_key creates a page with cloned block tree | `draft_config` matches layout blocks (regenerated IDs, same types/props) |
| Store without layout_key still works (default HeroBlock) | Backward compatibility |
| Store with unknown layout_key fails validation (422) | `layout_key` must be in allowed list |
| Block IDs are regenerated and differ from catalog originals | No ID collision with `config('designs.page_layouts')` |
| Non-owner cannot create page with layout | 403 |
| Applying a layout does not set published_config | Draft only (publish semantics preserved) |
| Applying a layout marks setup complete | Calls `markSiteSetupCompleted()` |
| Cross-tenant isolation when applying layouts | One tenant's layout page doesn't affect another |

Also update `DesignCatalogTest.php` to assert the new `page_type` + `industry` keys exist on every layout.
