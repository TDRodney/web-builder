# Fix the User-Flow Deadlock: Kit Application + Start From Scratch

## Problem

New users register → visit `/editor` → get redirected to `/designs` → see a preview-only page with no "Apply" button and no "Start from scratch" escape → **permanently stuck**. The only way out (`markSiteSetupCompleted()`) requires being in the editor, which is blocked by the redirect.

## Scope

Two new backend endpoints + frontend wiring in `DesignLibrary.vue`. No new pages, no new models, no new migrations.

> **Documentation sync rule (AGENTS.md):** Update `spec.md`, `AGENTS.md`, and `gaps.md` in the same change whenever the catalog schema or workflow changes. Every section below that touches the catalog, eligibility flow, or application endpoints must also update the corresponding documentation file.

## 1. Backend: `POST /designs/site-kits/{kit}/apply`

**New action class:** `app/Actions/Designs/ApplySiteKit.php`

**Logic inside `DB::transaction()`:**

1. **Authorize:** `auth()->id() !== $tenant->user_id` → 403
2. **Re-check eligibility (server-side):** `$tenant->isEligibleForInitialSiteKit()` → if false, 422 "Workspace is no longer eligible"
3. **Resolve kit:** look up `$kitKey` in `config('designs.site_kits')` → if missing, 404
4. **Resolve style:** look up `$siteKit['style_key']` in `config('designs.styles')`
5. **For each page in the kit's `pages` array:**
   - Resolve `layout_key` → `config('designs.page_layouts')[$layoutKey]['blocks']`
   - Deep-clone the block tree and regenerate every block `id` (recursive walk replacing each `id` with a fresh `uniqid('blk-')` value)
   - Create `Page::create()` with `draft_config` = cloned blocks, `published_config` = `null` (not published yet — matching AGENTS.md rule "Preserve publishing semantics"), `is_homepage` from kit def, `sort_order` from array index
6. **Apply style to tenant:** `$tenant->update(['theme_config' => $style['theme_config'], 'navigation_config' => $style['navigation_config'] ?? []])`
7. **Complete setup:** `$tenant->markSiteSetupCompleted()`
8. **Commit transaction**
9. **Return JSON:** `{ status: 'success', homepage_slug: <slug> }` so the frontend can redirect to `/editor?page=<homepage_slug>`

**Safety:** All writes are in one transaction. Any failure rolls back everything (no orphaned pages, no half-applied theme). The eligibility re-check inside the transaction prevents race conditions.

**Route:** `Route::post('/designs/site-kits/{kit}/apply', ...)->name('tenant.designs.apply-kit')` inside the auth middleware group, next to the existing `/designs` GET route.

**Controller:** Add `store` method to `TenantDesignLibraryController` that resolves the tenant, injects the `ApplySiteKit` action, and returns the response.

## 2. Backend: `POST /designs/start-from-scratch`

A lightweight endpoint that opts the user out of the kit flow.

**Logic:**

1. **Authorize:** ownership check → 403
2. **Re-check eligibility:** `$tenant->isEligibleForInitialSiteKit()` → if false, 422
3. **Complete setup:** `$tenant->markSiteSetupCompleted()`
4. **Return JSON:** `{ status: 'success' }`

No pages, theme, or navigation are created. The user enters the editor with a completely empty workspace and can create their first page normally (which will get the default HeroBlock from `TenantPageController::store`).

**Route:** `Route::post('/designs/start-from-scratch', ...)->name('tenant.designs.start-from-scratch')`

**Controller:** Add `startFromScratch` method to `TenantDesignLibraryController`.

## 3. Frontend: DesignLibrary.vue changes

Add two action buttons in the **aside status banner** area (lines 229-272), conditionally shown when `can_apply_site_kit` is true:

### "Use this design" button (Apply Kit)
- Shown when `can_apply_site_kit && selectedKitKey`
- Uses `useHttp` with a loading/disabled state
- On click: POST to the apply-kit route URL with `selectedKitKey`
- On success: `router.visit('/editor?page=' + homepage_slug)` (navigate to editor with the kit's homepage)
- On error: toast error message from response

### "Start from scratch" link
- Shown when `can_apply_site_kit`
- Small text button below the main CTA (not visually competing)
- Uses `useHttp` with loading state
- On click: POST to the start-from-scratch route
- On success: `router.visit('/editor')` (navigate to editor, no pages exist yet — user creates their first page)

## 5. Tests

### New feature test file: `tests/Feature/SiteKitApplicationTest.php`

| Test | What it verifies |
|---|---|
| Guests cannot apply a site kit | 401/redirect for unauthenticated POST |
| Non-owners cannot apply a site kit | 403 for wrong user |
| Applying a kit creates all pages from the manifest | Page count, slugs, titles, is_homepage match the kit definition |
| Kit-applied pages have draft_config but no published_config | `draft_config` is non-null, `published_config` is null |
| Block IDs are regenerated (not the catalog originals) | All block IDs differ from `config('designs.page_layouts')` originals |
| Theme and navigation are applied from the kit's style | Tenant `theme_config` and `navigation_config` match the style |
| Setup is marked complete after application | `site_setup_completed_at` is non-null |
| Kit cannot be applied twice | Second attempt returns 422 |
| Kit cannot be applied to a non-empty workspace | Returns 422 |
| Unknown kit key returns 404 | Safe failure |
| Transaction rollback on failure | If page creation fails midway, no pages are created, no theme applied |
| Cross-tenant isolation | One tenant's kit application does not affect another |
| Start from scratch marks setup complete | `site_setup_completed_at` is non-null |
| Start from scratch does not create pages | Page count remains 0 |
| Start from scratch does not set theme/navigation | Both remain null |
| Start from scratch cannot be called twice | 422 on second attempt |
